## Context

The Darkstar energy planner generates a schedule with `projected_soc_percent` values - a running simulation of where battery SoC will be at the end of each slot. Currently:

- **schedule.json** contains `projected_soc_percent` for future slots only (generated by Kepler solver at `planner/solver/adapter.py:511`)
- **slot_plans table** stores `planned_soc_percent` (the TARGET SoC, which differs from projected during multi-slot charge/discharge blocks)
- Historical slots have no `projected_soc_percent` in the database, preventing the UI from showing the projected line for past periods

The chart also has a visual inconsistency: Actual PV line uses `stepped: 'after'` while PV Forecast uses `tension: 0.4`.

## Goals / Non-Goals

**Goals:**
- Persist `projected_soc_percent` to `slot_plans` database for all slots (going forward only, no backfill)
- Enable SoC projection chart to display projected values across full 48-hour span
- Fix Actual PV chart line to render smoothly (matching PV Forecast style)
- Maintain backward compatibility with existing `planned_soc_percent` field

**Non-Goals:**
- Backfill historical data from old schedule.json files
- Modify the Kepler solver or projection algorithm
- Change any other chart line styles

## Decisions

### D1: Column Name and Type
**Decision**: Add `projected_soc_percent REAL NULL` to `slot_plans` table.

**Rationale**:
- `REAL` (SQLite float) matches existing `planned_soc_percent` column type
- `NULL` allows graceful handling of slots before migration
- Name clearly distinguishes from `planned_soc_percent` (target vs projection)

**Alternatives considered**:
- Renaming `planned_soc_percent` → `soc_target_percent` and adding new field: Rejected - breaks existing queries and adds migration complexity
- Using INTEGER: Rejected - loses decimal precision from simulation

### D2: Data Flow Architecture
**Decision**: Keep dual sources - `schedule.json` for future slots, database for historical.

**Rationale**:
- `schedule.json` remains SSOT for executor (already reads it every slot)
- Database provides historical data for charts
- No change to existing planner output flow

**Flow**:
```
Planner (Kepler) → schedule.json (future) → Executor
                           ↓
                  LearningStore.store_plan() → Database (historical)
                           ↓
                  schedule API → Frontend chart
```

### D3: Frontend Line Styling
**Decision**: Change Actual PV line from `stepped: 'after'` to `tension: 0.4`.

**Rationale**:
- Matches PV Forecast line style for visual consistency
- `tension: 0.4` provides smooth Bezier curves, appropriate for power readings
- Chart.js default, well-tested

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| Existing slots have NULL `projected_soc_percent` | Chart handles null gracefully (shows gap or skips). No backfill needed per requirements. |
| Migration adds column to active table | Alembic handles this safely. Backup created by docker-entrypoint.sh before migration. |
| Frontend may expect `projected_soc_percent` from schedule.json for future slots | Already handled - frontend pulls from schedule response which includes both sources |

## Migration Plan

1. **Pre-deploy**: Ensure docker-entrypoint.sh backup runs (already in place)
2. **Deploy**: Alembic migration runs automatically on container startup
3. **Verify**: Check `projected_soc_percent` column exists in slot_plans
4. **Rollback**: If needed, restore from `/data/backups/` backup file

Migration file: `alembic/versions/<hash>_add_projected_soc_percent_to_slot_plans.py`
